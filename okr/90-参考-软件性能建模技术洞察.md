# 软件性能建模技术洞察

## 1. 核心判断

当前 MUSA driver/API 性能建模应采用白盒软件模型。模型规则来自 `MUSA-Runtime` 和 `linux-ddk/musa` 源码，运行数据通过 MUPTI 采集，profiling 数据用于校准和验证。

不应把软件性能建模做成以下形式：

| 方式 | 问题 |
| --- | --- |
| 只看 API 总耗时 | 无法解释耗时来自 Runtime、Driver、同步等待、队列提交还是 KMD 边界 |
| 只做 kernel Top-K 对齐 | 只能证明 kernel 列表接近，不能证明 Driver/API 软件成本建模正确 |
| 只做黑盒拟合 | 无法定位源码模块，SDK 升级后模型容易失效 |
| 只做 trace replay | 可以重放时间线，但不能解释源码分支、状态转移和成本来源 |
| 用 printf/log 作为建模输入 | 热路径开销不可控，schema 不稳定，难以关联 API、command、submission、kernel |

推荐路线：

```text
源码规则
  -> 定义 API 状态转移和成本项

MUPTI 事件
  -> 采集 API、activity、relation、internal ModelEvent

关系重建
  -> 串起 Runtime API、Driver API、command、submission、kernel

成本分解
  -> 输出 Runtime、Driver、Memory、Stream、Command、Submit、Sync、IOCTL 分项

profiling 校准
  -> 校准 kernel/memcpy/sync 设备侧耗时和 Top50 kernel 排序

benchmark 验收
  -> 验证基础成本、插桩开销、事件完整性和跨版本稳定性
```

## 2. 当前已有建模思路

### 2.1 早期 trace replay 思路

早期方案以真实 trace 为输入，通过 replay 重建执行时间线。

```text
真实模型运行
  -> Runtime API trace
  -> Driver command trace
  -> profiler kernel trace
  -> 离线 link / replay / validator
  -> API/kernel/sync 热点和 predicted vs profiling 对齐
```

该思路适合做三类工作：

| 用途 | 说明 |
| --- | --- |
| trace schema 验证 | 验证采集字段是否足够串起 API、command、kernel |
| relation validator | 检查 Runtime API、Driver API、command、submission、kernel 是否能关联 |
| 离线对齐工具 | 生成 timeline，辅助分析 idle gap、sync wait、kernel 排序 |

该思路不适合作为主模型：

1. replay 依赖已经发生的 trace，预测能力弱。
2. replay 不能说明源码分支为什么发生。
3. replay 对未知组合、SDK 升级、新 workload 的泛化能力有限。
4. replay 只能解释“发生了什么”，不能完整解释“为什么耗时”。

因此，trace replay 应降级为验证工具。主路径应使用白盒软件模型。

### 2.2 当前白盒建模思路

当前 OKR 方案已经转向白盒建模。核心思想是：

```text
从源码提取状态机
  -> 用事件确认状态转移
  -> 用成本项解释耗时
  -> 用 profiling 校准设备侧执行
```

白盒建模的对象不是 GPU 内核算法本身，而是 Runtime/Driver 软件路径：

| 层级 | 建模内容 |
| --- | --- |
| Runtime | API wrapper、初始化、export table、module/function cache、Runtime 到 Driver 分发 |
| Driver API | 参数检查、context lookup、object lookup、Tools callback、错误码处理 |
| Memory | pool lookup、pool hit/miss、chunk allocate、mapping、free merge、peer map |
| Stream | dependency resolve、queue depth、async capacity wait、event wait、stream sync |
| Command | command create、build、merge、submit、wait、complete、error |
| Graph | instantiate、update、rebuild、node relation、graph launch |
| HAL/M3D | queue submit、cmd buffer、doorbell、DRM/KMD submit 边界 |
| MUPTI | API callback、activity、relation、ModelEvent 采集，不作为模型规则来源 |

该思路的关键变化是：MUPTI trace 不是模型本身。MUPTI 只提供证据、时间戳、关系和计数。模型规则来自源码。

### 2.3 当前推荐的组合思路

当前最合理的组合是：

```text
白盒 source rule
  + MUPTI ModelEvent
  + profiling activity
  + musa_benchmarks 校准
  + CTS 行为沉淀
```

各部分职责如下：

| 部分 | 职责 |
| --- | --- |
| source rule | 描述每个 API 的源码路径、状态转移、事件签名、成本项和异常分支 |
| ModelEvent | 记录源码内部关键状态和成本点 |
| activity | 提供 kernel、memcpy、memset、sync 等已有 profiling 事件 |
| relation builder | 重建 API、command、submission、kernel 的关系 |
| cost model | 根据 source rule 和事件数据计算分项耗时 |
| benchmark | 校准基础成本，验证插桩开销和事件签名 |
| CTS | 固化模型行为，支持跨 SDK 回归 |

### 3.2 NVIDIA CUPTI / Nsight Systems

CUPTI 的核心设计包括：

1. Runtime/Driver API callback。
2. GPU activity record。
3. activity buffer request/complete callback。
4. correlation ID 串联 API 与 kernel、memcpy、memset activity。
5. dropped record 查询和 flush 机制。
6. worker thread 处理 buffer delivery，降低对业务线程的影响。

Nsight Systems 在 CUPTI 之上提供系统级 timeline，覆盖 CUDA Runtime/Driver API、GPU workload、memory operation、kernel launch、CUDA Graph、CUDA Event、CUDA memory usage 等信息。它也明确提供 API 与 GPU workload 的双向关联能力，并通过“跳过低价值 API”“backtrace 阈值”“周期性 flush”等机制控制开销。

对当前方案的启示：

| CUPTI / Nsight 做法 | 对当前 MUPTI 方案的要求 |
| --- | --- |
| API callback 与 activity record 分层 | 保留 Runtime/Driver callback，同时把 kernel/memcpy/sync activity 与 internal ModelEvent 分开 |
| correlation ID 关联 API 与 GPU activity | `correlation_id` 必须贯穿 Runtime API、Driver API、command、submission、activity |
| activity buffer request/complete | MUPTI collector 必须有稳定 buffer lifecycle，而不是直接写日志 |
| dropped record 可查询 | `dropped_records`、buffer overflow、flush error 必须进入报告有效性判断 |
| 选择性 tracing | 支持 domain/event/filter，避免全量 tracing 干扰性能 |
| timeline 可视化 | 输出 Perfetto 或等价 timeline，有助于解释 queue、submit、sync wait |

差距：

1. 当前 MUPTI 已有 callback/activity 基础，但 internal ModelEvent 不完整。
2. 当前方案已提出 dropped record 和 flush，但还未成为验收门槛。
3. 当前报告缺少与 Nsight 类似的 timeline 输出要求。
4. 当前 relation 仍以 API/kernel 为主，需要补齐 command/submission 级 relation。

### 3.3 AMD ROCprofiler-SDK / rocprofv3

ROCprofiler-SDK 的设计与 CUPTI 接近：它提供 runtime API tracing callback，也提供 asynchronous activity records。`rocprofv3` 面向应用级 tracing 和 profiling，能采集 kernel execution、memory、API call、hardware event，并可输出 JSON/CSV 或 Perfetto 兼容 trace。

对当前方案的启示：

| ROCm 做法 | 对当前 MUPTI 方案的要求 |
| --- | --- |
| callback API + activity API | 当前 MUPTI 需要形成 callback、activity、ModelEvent 三层数据模型 |
| rocprofv3 面向工具化采集 | 需要独立 collector CLI，而不是只保留 demo 或日志 |
| Perfetto 兼容输出 | 建议增加 `trace.perfetto` 或 `trace.pftrace` 输出 |
| kernel、memory、sync、API 同一 timeline | 当前报告应把 API、command、activity、sync wait 合并展示 |
| thread trace / PC sampling 用于深度硬件分析 | 本 OKR 不应把硬件 thread trace 放入主模型，主模型聚焦软件路径 |

差距：

1. 当前方案强调白盒源码规则，这是 ROCprofiler 普通 trace 不直接提供的能力。
2. 当前 MUPTI collector 尚未达到 rocprofv3 这类工具化程度。
3. Perfetto 兼容输出应加入工程化交付，而不是只输出 JSONL/Parquet。

### 3.4 Intel Level Zero / VTune / ITT

Level Zero Tools Programming Guide 提供 API tracing loader layer，工具可以创建 tracer、注册 prologue/epilogue callback，并动态启停 tracing。ITT API 通过 domain、task begin/end、task id、parent id、metadata 描述用户代码中的逻辑任务。VTune GPU Offload Analysis 会区分 CPU/GPU 任务、GPU idle、API trace、host stack、数据传输和 GPU 利用率，并提醒 API tracing 会影响 CPU 侧性能。

对当前方案的启示：

| Intel 做法 | 对当前 MUPTI 方案的要求 |
| --- | --- |
| prologue/epilogue callback | API enter/exit、ModelEvent begin/end 应保持成对语义 |
| dynamic enable/disable | MUPTI tracing 需要运行时 domain 开关 |
| ITT domain/task/id | ModelEvent 应有 domain、event_id、relation_id、parent_id |
| metadata 绑定 task | source rule、状态变量、payload 字段应结构化 |
| CPU/GPU offload 分解 | 报告必须区分 host API cost、sync wait、GPU execution |
| tracing 影响 CPU 性能 | 插桩开销必须成为正式验收项 |

差距：

1. 当前方案已有 domain/event 概念，但没有明确 `parent_id` 和 task/span 结构。
2. 当前 source rule 需要绑定 event domain，形成稳定命名。
3. 当前报告需要把 CPU-side API trace 和 GPU-side activity 分开展示，再通过 relation 连接。

### 3.5 Perfetto

Perfetto 是通用 timeline tracing 系统，强调低开销采集、跨进程统一 trace、离线分析和浏览器可视化。它的事件模型对当前方案有直接借鉴价值：

| Perfetto 概念 | 当前方案对应 |
| --- | --- |
| slice | internal span，例如 command build、submit、sync wait |
| counter | queue depth、pool used bytes、async count、drop count |
| flow | Runtime API -> Driver API -> command -> submission -> kernel |
| track | thread、stream、context、command queue |
| trace processor / SQL | 离线 relation builder 和 cost breakdown |

对当前方案的启示：

1. ModelEvent 不应只定义 begin/end，还应明确 span、counter、instant、relation 四类。
2. `queue_depth`、`merge_list_size`、`pool_used_bytes`、`drop_count` 等应作为 counter 事件。
3. `Runtime API -> kernel activity` 这种跨线程、跨队列关系应使用 flow/relation 表达。
4. 输出 Perfetto trace 可以显著降低评审和问题定位成本。

### 3.6 OpenTelemetry

OpenTelemetry 面向分布式系统观测，核心是 trace、span、span context、attribute、span event、collector/exporter。它不适合直接作为 MUSA Driver 热路径采集格式，但适合作为长期 schema 设计参考。

可借鉴点：

| OpenTelemetry 概念 | 当前方案对应 |
| --- | --- |
| trace id | workload/run id |
| span id | API span、command span、submit span |
| parent span | Runtime API -> Driver API -> internal span |
| span attributes | api name、stream id、command type、size、status |
| span event | 状态点，例如 pool miss、merge flush、error |
| collector/exporter | MUPTI collector 和 report pipeline |
| semantic attributes | 统一字段命名和版本兼容 |

限制：

1. OpenTelemetry 的通用 SDK 不适合放在 Driver 热路径。
2. 可以离线把 MUPTI trace 转为 OTLP 或类 OTLP 表，用于跨系统报告。
3. 采集层仍应使用低开销二进制事件或 MUPTI activity buffer。

### 3.7 eBPF / USDT

USDT 是用户态静态 tracepoint 机制，tracepoint 编译后通常以 NOP 形式存在，工具通过 eBPF/uprobe attach 到这些位置。USDT semaphore 可在有工具附着时才收集高成本参数。

对当前方案的启示：

| USDT / eBPF 做法 | 对当前方案的意义 |
| --- | --- |
| 静态 tracepoint 默认低成本 | ModelEvent 默认关闭时只保留轻量判断 |
| semaphore 控制参数收集 | 高成本 payload 只在 tracing 开启时生成 |
| uprobe 可附着用户态函数 | 可作为不改源码时的补充验证手段 |
| 可观测 syscall/KMD 边界 | 可辅助分析 ioctl、futex、poll 等系统边界 |

限制：

1. eBPF/USDT 适合系统边界和辅助验证，不适合作为主建模通道。
2. Driver 内部对象关系复杂，直接用 uprobe 解析参数成本高、稳定性弱。
3. 当前源码可改，优先使用 MUPTI private hook 和结构化 ModelEvent。

### 3.8 业界共性设计

主流方案存在以下共性：

| 共性设计 | 说明 | 当前方案状态 |
| --- | --- | --- |
| API callback | 捕获 Runtime/Driver API enter/exit | 已具备基础 |
| activity record | 捕获异步 GPU work、memcpy、sync | 已具备部分能力 |
| correlation / flow | 串联 host API 与 device activity | 已有 correlation，需扩展 command/submission relation |
| buffer callback | 工具提供 buffer，采集层批量写入 | 已有 activity buffer，应扩展 internal event |
| dropped record | 事件丢失可观测 | 已提出，需进入验收 |
| dynamic enable | 按 domain/event 控制 | 已提出，需实现 |
| low overhead | 默认关闭、过滤、采样、后台 flush | 已提出，需 benchmark 验证 |
| timeline output | timeline 可视化和离线分析 | 当前不足 |
| semantic schema | 稳定字段命名和版本 | 当前不足 |
| validation suite | 用 benchmark 验证开销和覆盖 | 已提出，需落地 |

### 3.9 与当前方案的直接对照

| 能力 | CUPTI / Nsight | ROCprofiler / rocprofv3 | Level Zero / VTune / ITT | Perfetto / OTel / eBPF | 当前 MUPTI 方案 | 结论 |
| --- | --- | --- | --- | --- | --- | --- |
| API tracing | CUDA Runtime/Driver API | HIP/HSA runtime API | Level Zero/SYCL/OpenCL API | OTel span / USDT probe | Runtime/Driver Tools callback | 方向一致 |
| GPU activity | kernel、memcpy、memset、sync | kernel、memory、hardware event | GPU offload task | timeline slice | kernel/memcpy/sync tracepoint | 需要完善 activity 支持矩阵 |
| 关联机制 | correlation ID | correlation/activity record | task id/parent id | flow/span context | correlation id | 需要补 `command_id`、`submission_id`、`parent_id` |
| 内部软件状态 | 有限，偏工具视角 | 有限，偏工具视角 | task/domain 可表达用户逻辑 | 可表达 slice/counter/flow | 计划新增 ModelEvent | 这是当前方案的核心差异点 |
| buffer/flush | activity buffer callback、flush、dropped | buffered tracing、flush | 工具采集 | trace daemon / collector | activity buffer 基础 | internal ModelEvent 也要进入 buffer 管理 |
| 开销控制 | 过滤、跳过低价值 API、flush 策略 | tracing option | dynamic enable | semaphore/sampling | ready flag、domain 开关设想 | 必须用 benchmark 验收 |
| 可视化 | Nsight timeline | Perfetto/CSV/JSON | VTune timeline | Perfetto UI / backend | 当前主要是 JSONL/Parquet 设想 | 应增加 Perfetto 输出 |
| schema 稳定 | activity struct version | SDK schema | API/domain/task | semantic attributes | ModelEvent header 设想 | 需要版本化 event schema |
| 白盒源码规则 | 不直接提供 | 不直接提供 | 不直接提供 | 不直接提供 | source rule | 当前方案优势，必须保留 |

### 3.10 对当前方案的修订要求

结合业界方案，当前文档和 OKR 方案需要增加以下硬性要求：

1. `ModelEvent` 事件类型固定为 `span`、`instant`、`counter`、`relation` 四类。
2. 除 `correlation_id` 外，统一引入 `span_id`、`parent_id`、`command_id`、`submission_id`、`activity_id`。
3. collector 输出必须包括 `trace.jsonl`、`relations.parquet`、`api_cost_breakdown.parquet`，并增加 Perfetto 兼容 trace。
4. dropped record、buffer overflow、flush error 必须进入报告有效性判断。
5. 插桩开销必须通过 benchmark 验收，至少覆盖 trace off、API-only、internal targeted 三种模式。
6. source rule 必须绑定 ModelEvent domain 和 event id，不能只写自然语言路径。
7. 高成本 payload 必须延迟生成，只有 tracing 开启并命中 domain/filter 时才采集。
8. eBPF/USDT 作为 ioctl/KMD 边界和系统调用辅助验证，不替代 MUPTI 主通道。
9. 报告应增加 timeline 视图，用于展示 API、stream、command、submission、kernel 和 sync wait。
10. 长期 schema 应参考 OTel 的 attribute 命名方式，但采集热路径不直接使用 OTel SDK。

业界对标后的最终判断：

```text
当前 MUPTI + ModelEvent + source rule 方向正确。
它与 CUPTI / ROCprofiler / Level Zero tracing 的主流结构一致。
当前最大缺口不是方向，而是工程化程度：
  1. internal event schema 未稳定；
  2. collector 未工具化；
  3. relation builder 未落地；
  4. Perfetto/timeline 输出不足；
  5. dropped record 和 overhead 尚未成为硬验收。
```

## 4. 当前已有建模方法

### 4.1 source rule 方法

source rule 是软件模型的核心。每个 Top90 Driver API 都要有独立规则。

推荐格式：

| 字段 | 内容 |
| --- | --- |
| API | Driver API 名称 |
| 入口 | wrapper 文件和 `muapi*` 实现 |
| 核心路径 | 从 Driver API 到 Core/HAL/M3D 的完整函数链 |
| 状态变量 | context、stream、event、memory pool、command、submission、semaphore |
| 状态转移 | created、queued、built、submitted、completed、error 等状态变化 |
| 事件签名 | 必须出现的 API event、ModelEvent、activity、relation |
| 成本项 | API 拆成哪些可解释耗时 |
| 异常分支 | invalid handle、pool miss、sync wait、submit error、graph rebuild 等 |
| 验证用例 | 触发该路径的 benchmark 或模型片段 |

示例：`muLaunchKernel`

```text
入口：
  muLaunchKernel wrapper
  muapiLaunchKernel

核心路径：
  muapiLaunchKernel
  -> Stream::CmdLaunchKernel
  -> Context::ResolveDependencyAndQueueCommand
  -> Stream::QueueCommand
  -> Stream::AsyncSubmit
  -> DispatchCommand::Build
  -> DispatchCommand::Submit
  -> Command::SubmitToQueue
  -> HAL Queue::Submit
  -> M3D Queue::SubmitInternal
  -> DRM Queue::OsSubmit
  -> mtgpu Device::SubmitCommands / SubmitCommandsWithDoorbell

成本项：
  T_driver_wrapper
  T_validation
  T_function_lookup
  T_stream_lookup
  T_command_create
  T_dependency_resolve
  T_queue_command
  T_command_build
  T_merge_check
  T_submit_to_queue
  T_hal_m3d_submit
```

示例：`muMemAlloc`

```text
核心路径：
  muapiMemAlloc
  -> context / device lookup
  -> memory pool lookup
  -> pool allocate
  -> pool hit 或 pool grow
  -> HAL memory allocation / map
  -> peer map

成本项：
  T_context_lookup
  T_pool_lookup
  T_pool_lock
  T_pool_hit
  T_pool_grow
  T_hal_alloc
  T_map
  T_peer_map
```

示例：`muStreamSynchronize`

```text
核心路径：
  muapiStreamSynchronize
  -> stream lookup
  -> Stream::WaitFinish
  -> Command::Wait 或 Stream::AsyncWait
  -> semaphore wait
  -> engine last error query
  -> user pool update

成本项：
  T_stream_lookup
  T_last_command_lookup
  T_sync_wait
  T_engine_error_query
  T_pool_update
```

### 4.2 状态机方法

软件性能模型需要维护源码中的关键状态。状态不完整时，模型只能拟合总耗时，无法解释分支。

建议维护以下状态：

```text
RuntimeState:
  initialized
  export_table_ready
  current_device_per_thread
  module_cache
  function_cache

DriverState:
  current_context
  context_map
  stream_map
  event_map
  graph_map

MemoryState:
  pool_map
  free_buckets
  chunks
  allocated_objects
  fragmentation

StreamState:
  queue_depth
  dependency_edges
  async_count
  inflight_count
  merge_list

CommandState:
  created
  queued
  built
  submitted
  completed
  error

SubmissionState:
  submission_id
  wait_semaphores
  signal_semaphores
  doorbell
  ioctl_status
```

状态机模型的价值：

| 能力 | 说明 |
| --- | --- |
| 区分路径 | pool hit 和 pool grow 走不同路径，不能用同一个平均值 |
| 解释等待 | sync wait、async capacity wait、dependency wait 必须拆开 |
| 解释提交 | command merge、merge flush、submit count 会影响 host submit 成本 |
| 支持版本对比 | SDK 变化可以定位到状态转移或成本项变化 |

### 4.3 事件采集方法

当前可复用两类现有事件，再新增一类内部事件。

```text
第一类：Tools callback
  Runtime API / Driver API enter-exit

第二类：Driver MUPTI tracepoint
  kernel / memcpy / memset / sync / graph / submission relation

第三类：ModelEvent
  memory / stream / command / graph / sync / ioctl 内部成本点
```

ModelEvent 建议使用 private hook 起步：

```text
MUSA-Runtime / Driver / Core / HAL
  -> EmitModelEvent
  -> MUPTI collector
  -> activity buffer 或独立 internal buffer
  -> 离线 parser
```

事件设计要求：

| 要求 | 说明 |
| --- | --- |
| 默认关闭 | 未开启 collector 时只读 atomic ready flag |
| 固定结构 | header 固定，payload 按 event id 解码 |
| 成对记录 | 耗时事件使用 begin/end |
| 关系显式 | API、command、submission、kernel 关系单独记录 |
| 不写日志 | 热路径不做字符串格式化、不写文件、不加锁 |
| 可采样 | 高频事件支持 domain/event 开关 |

### 4.4 关系重建方法

关系重建是连接软件模型和 profiling 结果的关键。

需要重建的关系：

```text
Runtime API
  -> Driver API
  -> Core command
  -> submission / kick
  -> kernel / memcpy / sync activity
```

主要关联字段：

| 字段 | 用途 |
| --- | --- |
| `correlation_id` | 关联 API、command、kernel activity |
| `stream_id` | 重建 stream 顺序和并发 |
| `context_id` | 区分设备和上下文 |
| `command_id` | 关联 command 生命周期 |
| `submission_id` | 关联 command 与 HAL/M3D submit |
| `kernel_name` | 与 profiler kernel 结果对齐 |

输出关系表：

```text
relations:
  runtime_api_id
  driver_api_id
  command_id
  submission_id
  activity_id
  relation_type
  confidence
```

关系完整性必须成为验收指标。只比较 Top50 kernel overlap 不足以证明模型正确。

### 4.7 验证方法

验证不应只比较端到端时间。

| 验证项 | 标准 |
| --- | --- |
| API 覆盖 | 累计耗时 Top90% Driver API 有 source rule、事件签名和成本分项 |
| 关系完整性 | API -> command -> submission -> kernel 关系重建召回率不低于 95% |
| Top50 kernel | 按累计耗时排序的 Top-K overlap 不低于 90% |
| 成本分项 | API self、memory、stream、command、submit、sync wait、ioctl 分项可解释 |
| 状态路径 | pool hit/miss、command merge、graph rebuild、sync wait 与事件一致 |
| 事件质量 | dropped record、buffer overflow、flush error 必须输出 |
| 插桩开销 | trace off 在 benchmark 噪声内；trace on 开销有量化结果 |
| 版本对比 | SDK 差异能定位到源码模块、状态转移或成本项 |

## 7. 推荐落地方法

### 7.1 第一阶段：建立最小闭环

覆盖三个 API：

```text
muLaunchKernel
muMemAlloc / muMemFree
muStreamSynchronize
```

必须交付：

| 交付物 | 内容 |
| --- | --- |
| source rule | 三个 API 的源码路径、状态转移、成本项 |
| ModelEvent MVP | queue、build、submit、pool hit/miss、sync wait |
| collector MVP | 输出 API、activity、ModelEvent、relation |
| relation report | Runtime API -> Driver API -> command -> submission -> kernel |
| cost breakdown | 每个 API 的分项耗时 |
| overhead report | trace off / trace on 开销 |

### 7.2 第二阶段：扩展 Top90 API

流程：

```text
3 个模型运行
  -> 采集 API 和 kernel 基线
  -> 统计 Top90 Driver API
  -> 为 Top90 API 建 source rule
  -> 按模块补 ModelEvent
  -> 生成 API cost breakdown
  -> 校准 Top50 kernel overlap
```

Top90 API 不应逐个临时插桩。应按模块补事件：

| 模块 | 事件 |
| --- | --- |
| memory | pool lookup、hit/miss、grow、map、free merge |
| stream | dependency、queue depth、async capacity wait、sync wait |
| command | build、merge check、merge flush、submit、complete |
| graph | instantiate、node process、update、rebuild、launch |
| module | module load、function lookup、cache hit/miss |
| HAL/M3D | queue submit、cmd buffer、doorbell、KMD submit |

### 7.3 第三阶段：工程化和 CTS

工程化重点：

| 方向 | 内容 |
| --- | --- |
| schema | ModelEvent header、domain、event id、payload 版本化 |
| collector | 配置化采集、buffer 压测、flush、异常退出处理 |
| model | source rule engine、relation builder、cost fitter |
| report | API 热点、kernel 热点、源码瓶颈、优化建议 |
| benchmark | calibration、overhead、event signature |
| CTS | dense/MoE 行为特征、memory/stream/graph/sync 事件签名 |

CTS 不应沉淀原始模型 trace，应沉淀可复现的软件行为：

```text
memory pool hit/miss
stream dependency
event wait
command merge / no merge
graph instantiate / update / launch
sync wait reason
KV cache allocation pattern
attention / MLP launch pattern
expert routing small-kernel pattern
```

## 8. 报告输出标准

每个模型的性能建模报告应包含：

| 章节 | 内容 |
| --- | --- |
| workload 信息 | 模型、场景、batch、seq length、SDK version、commit、设备 |
| API 热点 | Top Runtime API、Top Driver API、host self、inclusive、sync wait |
| kernel 热点 | Top50 kernel、累计耗时、Top-K overlap |
| relation 质量 | API -> command -> submission -> kernel recall、未关联项 |
| 成本分项 | Runtime、Driver、Memory、Stream、Command、Submit、Sync、IOCTL |
| 源码归因 | 对应 source rule、文件、函数、状态变量 |
| 异常路径 | pool miss、merge flush、sync wait、submit delay、event drop |
| 优化建议 | 明确到模块和路径，不只写“优化 Driver” |
| 事件质量 | dropped record、buffer overflow、flush error |
| 插桩开销 | trace off/on 对比 |

| API | total_us | runtime_us | driver_us | memory_us | queue_us | build_us | submit_us | sync_wait_us | bottleneck |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | --- |

## 9. 技术风险

| 风险 | 表现 | 处理 |
| --- | --- | --- |
| 把 profiling 当模型 | 只能解释 kernel，不解释 Driver 软件路径 | 明确模型规则来自源码 |
| sync wait 污染 host cost | `muStreamSynchronize` 被误判为 Driver CPU 开销 | sync wait 单独成项 |
| relation 不完整 | kernel 无法关联到 API 或 command | correlation、command_id、submission_id 必须统一 |
| 事件过多影响性能 | tracing 改变被测系统行为 | 默认关闭、domain 开关、TLS buffer、采样 |
| 单 subscriber 冲突 | 无法和 profiler 同时使用 | collector 独占或 collector 合并 activity |
| ABI 不稳定 | SDK 升级后 collector 失效 | private hook 先验证，schema 版本化 |
| source rule 维护成本高 | API 行为变化后模型失效 | source rule 绑定 commit，CI 检查事件签名 |

## 10. 最终判断

当前已有资料已经完成了三项基础工作：

1. 明确了白盒建模方向，否定了纯黑盒 replay 作为主方案。

3. 打通 MUPTI ModelEvent private hook 和 collector。
4. 输出第一版 `api_cost_breakdown`、`relation_recall_report`、`overhead_report`。

完成这四件事后，软件性能建模才从技术方案进入可验证实践阶段。
