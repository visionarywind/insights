# My Top 5 Things

## My OKR

- **Driver/API白盒性能建模**：通过 MUPTI / ModelEvent 对 Driver 内部链路进行可解释拆解，支持 API cost breakdown 和模型 Top90 API 覆盖。
- **Driver版本性能看护**：在版本发布前通过 Perf / profiling 量化评估 Driver 性能表现，定位 Runtime / Driver / HAL / M3D 边界的软件瓶颈。
- **行为CTS与回归沉淀**：将稳定的模型行为模式沉淀为 CTS 或 benchmark，用于 SDK / Driver 版本差异分析和性能回归看护。

## Executive Summary

[DriverPerf/M0] 已整理 Driver Perf Roadmap，明确核心目标是版本发布前量化评估 Driver 性能，并通过 Driver 内部事件定位性能瓶颈；当前文档已落到 `insights/okr/Roadmap.md`。

[DriverPerf/ModelEvent] 已明确白盒建模拆解口径，将一次 Runtime / Driver API 调用拆成 `runtime wrapper`、`driver wrapper`、`object/context lookup`、`memory/stream/command processing`、`dependency/queue/build`、`submit/HAL/M3D/OS boundary` 等阶段，用于后续 API cost breakdown。

[DriverPerf/Roadmap] 已按照当前计划固定 M0-M4 路线：M0 跑通内存 API 链路，M1 覆盖单模型 Top90 API，M2 抽象主流模型行为模式，M3 与 msight/profiling 对齐，M4 沉淀 CTS 和性能看护。

[DriverPerf/M0] 已将 M0 原型收敛到 `musaMalloc / muMemAlloc -> musaFree / muMemFree`，当前阶段不追求全量 API 覆盖，优先证明 Runtime API / Driver API / Driver 内部 ModelEvent 到 cost breakdown 的链路可闭环。

[DriverPerf/Risk] 当前主要风险是 workload 尚未固定、M0 ModelEvent 字段尚未冻结、profiling 对齐依赖后续工具链；下周需要先选择一个模型 workload，并为内存 API 定义最小事件字段和验收口径。

# 未来计划

## 1. 推进 M0 内存 API 原型

围绕 `musaMalloc / muMemAlloc -> musaFree / muMemFree` 输出 source rule、最小 ModelEvent 字段、采集链路说明和初版 cost breakdown 字段设计。

## 2. 固定 M1 模型候选

明确模型类型、batch size、sequence length、采集方式和 Top90 API 统计口径，为 M1 单模型 Top90 API 覆盖提供稳定输入。

## 3. 补齐 M0/M1 验收口径

确认是否能采集 API 和 Driver 内部事件、是否能输出阶段耗时、是否能说明 unknown cost、是否能支撑后续 msight / profiling 对齐。
