# Driver/API 白盒性能建模周报

周期：2026-05-25 ~ 2026-05-29

## 1. 本周进展

### 1.1 明确目标

本周明确了 Driver/API 白盒性能建模的核心目标：

- 在版本发布前，通过 Perf / profiling 手段量化评估 Driver 性能表现。
- 通过采集 Driver 内部事件，定位影响整体性能的关键瓶颈。
- 将 Runtime / Driver API 调用拆成可解释的内部链路，输出阶段级耗时分解。

当前目标已经整理到 [Roadmap.md](Roadmap.md)。

### 1.2 梳理整体技术路线

确定整体思路为：通过 MUPTI 埋点，将 Driver 执行过程串成可解释的白盒性能模型。

当前拆解口径为：

```text
API total time
  = runtime wrapper
  + driver wrapper
  + object / context lookup
  + memory / stream / command processing
  + dependency / queue / build
  + submit / HAL / M3D / OS boundary
```

该口径用于后续输出 API cost breakdown，并支撑版本发布前的性能看护。

### 1.3 确定阶段计划

本周按照当前计划确定了 M0-M4 阶段划分：

| 阶段 | 目标 |
| --- | --- |
| M0 | 针对内存 API 跑通链路 |
| M1 | 确定模型，并覆盖该模型的 Top90 API |
| M2 | 抽象主流模型行为模式，并验证行为级性能建模 |
| M3 | 使用 msight / profiling 对齐性能建模结果 |
| M4 | 沉淀到 CTS，并形成性能看护 |

其中 M0 明确以 `musaMalloc / muMemAlloc -> musaFree / muMemFree` 为原型链路，不在该阶段追求全量 API 覆盖。

## 2. 当前结论

当前阶段不应直接铺开 Top90 API 或模型行为 CTS，优先级应放在 M0 内存 API 链路原型上。

M0 的目标是证明以下链路能够闭环：

```text
Runtime API / Driver API
  -> Driver 内部 ModelEvent
  -> MUPTI collector
  -> cost breakdown
```

只要能解释一次内存 API 调用在 Driver 内部经过了哪些阶段、每个阶段耗时多少，就可以支撑后续扩展到模型 Top90 API。

## 3. 风险与问题

| 风险 | 影响 | 处理建议 |
| --- | --- | --- |
| workload 尚未固定 | M1 Top90 API 无法稳定统计 | 下周需要先选择一个模型和固定运行参数 |
| M0 事件字段还未冻结 | collector 和 cost breakdown 后续可能反复修改 | 先为内存 API 定义最小 ModelEvent 字段 |
| 过早扩展 Top90 API | 容易导致链路未闭环时范围失控 | M0 只做内存 API 原型，M1 再进入模型 Top90 |
| profiling 对齐依赖工具链 | M3 前置条件较多 | M0/M1 先保留对齐字段，如 correlation、timestamp、api name |

## 4. 下周计划

### 4.1 M0 内存 API 原型

围绕以下链路做最小闭环：

```text
musaMalloc / muMemAlloc
  -> Driver 内部 memory path
  -> musaFree / muMemFree
```

计划输出：

- 内存 API source rule。
- 内存 API ModelEvent 最小字段。
- Runtime API / Driver API / ModelEvent 的采集链路说明。
- 初版 API cost breakdown 字段设计。

### 4.2 固定 M1 模型候选

选择一个模型 workload 作为 M1 原型对象，并明确：

- 模型类型。
- batch size / sequence length 等运行参数。
- 采集方式。
- Top90 API 统计口径。

### 4.3 补齐验收口径

为 M0/M1 补充可验收指标：

- 是否能采集 API 和 Driver 内部事件。
- 是否能输出阶段耗时。
- 是否能说明 unknown cost。
- 是否能支撑后续 msight / profiling 对齐。
