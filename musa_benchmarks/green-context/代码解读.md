## 一、代码定位

这是一个 **MUSA Green Context 验证测试程序**，参考 MUSA Programming Guide §4.6。核心目标：验证 GPU SM（Streaming Multiprocessor）分区隔离功能——即通过 `muDevSmResourceSplitByCount` 将 GPU 的 SM 划分为独立分区，然后用 Green Context 绑定各分区，验证分区后的 kernel 执行互不干扰。

## 三、两个测试详解

### Test 1: SM Provisioning + Kernel Correctness（第 185-276 行）

**当前被注释掉**（第 664 行），但不影响理解其意图：

```
流程：
1. muDeviceGetDevResource  → 获取全部 SM 资源
2. muDevSmResourceSplitByCount → 分区：critical(16SM) + remainder(剩余)
3. muGreenCtxCreate → 为每个分区创建 Green Context + Stream
4. muGreenCtxGetDevResource → 验证 provisioned SM 数与分区一致
5. 在 critical Context 上运行 vector_add_kernel → 验证结果正确
6. destroy_green_context → 销毁资源
```

**验证点**：
- SM 分区是否精确（provisioned SM 数 == split result）
- Green Context 上的 kernel 是否能正确执行

### Test 2: Critical Execution Isolation（第 303-635 行）

**当前实际运行的测试**，验证 Green Context 的隔离性能：

```
核心设计：
  ┌─────────────────────────────────────────────┐
  │ GPU 总 SM = N                               │
  │  ├─ Critical 分区 (16 SM) ← Green Context A │
  │  │   运行 critical_kernel (6M spin cycles)   │
  │  └─ Bulk 分区      (剩余 SM) ← Green Context B  │
  │      运行 delay_kernel (25M spin cycles)     │
  └─────────────────────────────────────────────┘

对比场景：
  Baseline: delay_kernel + critical_kernel 都在 Primary Context 上（无 SM 隔离）
  Green:    delay_kernel 在 Bulk GC，critical_kernel 在 Critical GC

预期：Green Context 下 critical_kernel 受 bulk 干扰更少 → 整体延迟(baseline - green) 更短
```

**关键 kernel 设计**：
- `delay_kernel` 和 `critical_kernel` 都声明 **192 KiB 静态 shared memory**，确保每个 block 独占一个 MP（单 MP 不足以同时容纳两个这样的 block）
- `delay_kernel` 通过 `started_flag` 做 host 端同步，确保 critical launch 发生在 delay 已占满所有 bulk SM 之后
- 用 `clock64()` 做 busy-wait spin，精确控制 kernel 执行时长

**判定逻辑**（第 614-633 行）：
```cpp
// 三选一的 PASS 条件：
if (gc_total_ms < baseline_total_ms * 0.8f && improvement > 2.0f)    → PASS (缩短 >20% 且 >2ms)
if (improvement > 5.0f && gc_total_ms < baseline_total_ms)           → PASS (缩短 >5ms)
if (rel_improvement > 0.15f && gc_total_ms < baseline_total_ms)      → PASS (缩短 >15%)
// 否则 FAIL
```

**Benchmark 设计**：
- 默认 `bench_iters=20`，取平均值
- 默认 `warmup=1`（可用 `skip-warmup` 跳过）
- 每次 benchmark 前先跑 isolated kernel timing（solo 模式，无 peer 干扰），帮助校准基线

## 四、关键 API 使用方式

```cpp
// 1. 获取全部 SM 资源
MUdevResource all_sm{};
muDeviceGetDevResource(dev, &all_sm, MU_DEV_RESOURCE_TYPE_SM);

// 2. 分区：从 total 中切出 critical_target 个 SM
unsigned int nb_groups = 1;
MUdevResource critical_sm{}, remainder{};
muDevSmResourceSplitByCount(&critical_sm, &nb_groups, &all_sm, &remainder, 0, critical_target);

// 3. 生成资源描述符并创建 Green Context
MUdevResourceDesc desc{};
muDevResourceGenerateDesc(&desc, &critical_sm, 1);
MUgreenCtx green_ctx;
muGreenCtxCreate(&green_ctx, desc, dev, MU_GREEN_CTX_DEFAULT_STREAM);

// 4. 从 Green Context 获取普通 MUcontext 并设为 current
MUcontext ctx;
muCtxFromGreenCtx(&ctx, green_ctx);
muCtxSetCurrent(ctx);

// 5. 创建 Green Context 绑定的 Stream
MUstream stream;
muGreenCtxStreamCreate(&stream, green_ctx, MU_STREAM_NON_BLOCKING, 0);

// 6. 验证 provisioning
MUdevResource actual_sm{};
muGreenCtxGetDevResource(green_ctx, &actual_sm, MU_DEV_RESOURCE_TYPE_SM);
// actual_sm.sm.smCount 应 == critical_sm.sm.smCount

// 7. Cleanup
muStreamDestroy(stream);
muGreenCtxDestroy(green_ctx);
```

## 五、设计模式与工程细节

| 模式 | 说明 |
|------|------|
| **Host-GPU 同步** | `wait_delay_started_flag` 轮询 `musaHostAllocMapped` 的 flag，确保 critical launch 发生在 delay kernel 已就绪之后 |
| **MP 独占** | 192 KiB static smem 确保每个 block 独占一个 MP，避免 delay/critical block 被调度到同一 MP |
| **DCE 防护** | `occupy_block_smem_192k` 用 `noinline` + `volatile` 触摸每个 byte 并返回 tag，防止编译器优化掉共享内存分配 |
| **误差消除** | warmup + 多轮平均 (20 iters) + isolated timing calibration |
| **RAII 风格清理** | `GreenContextBundle` + `destroy_green_context` 确保资源释放 |

## 六、适用场景

- **GPU 虚拟化 / MIG 场景**：验证 SM 分区隔离是否生效
- **性能 SLA 保障**：为高优先级 kernel 预留专用 SM
- **Driver/KMD 回归测试**：升级后验证 Green Context 功能无损
